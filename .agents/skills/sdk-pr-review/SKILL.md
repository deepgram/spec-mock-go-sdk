# Reviewing an auto-regen PR

PRs from the fan-out workflow in `deepgram/spec` arrive here with `api/`
regenerated by the codegen plugin and (within seconds) `pkg/` updated by
`spec-idiomatic`. Your job as a reviewer is not to re-do the bot's work
— it's to verify the bot did the right things in the right places.

The universal facade-author rules the bot follows (3-tier model,
converter naming, additive rule, facade-mirror rule, wire-test contract)
live in
[`spec-idiomatic/prompts/system.md`](https://github.com/deepgram/spec-idiomatic/blob/main/prompts/system.md).
Read that if you want to know what the bot is supposed to do; this
skill is what you check.

## The standard PR shape

A clean auto-regen PR looks like this:

1. **Initial commit (from `deepgram/spec`'s fan-out workflow):**
   modifies files under `api/` only. Generated by smithy.
2. **One or more bot commits (from `spec-idiomatic-bot`):**
   modify files under `pkg/`, plus optionally `Example_*` tests,
   `README.md` opener, `llms.txt`.
3. **Deterministic cleanup-pass commits (also `spec-idiomatic-bot`):**
   tier-2 facade removals, `@deprecated` comment propagation, the
   `BREAKING_CHANGES.md` rewrite. Commit subjects carry
   `Spec-Idiomatic-Pass: tier2-cleanup` / `deprecated-propagation`
   trailers.
4. **PR body:** includes the non-conditional work-done checklist.
5. **CI:** `go build ./...` and `go test ./...` both green.

Anything outside this shape needs explanation in the PR description.

## Review checklist

### Pipeline integrity

- [ ] No file under `api/` is missing its
  `// Code generated by smithy-go-codegen DO NOT EDIT.` header.
- [ ] No human-authored commits to `api/`. If a human touched anything
  in `api/`, it will be wiped by the next regen — push back.
- [ ] The PR head is a bot facade-update commit (or a deterministic
  cleanup-pass commit), not the raw generated-only commit. If the bot
  didn't run, find out why before merging.

### Facade absorption

- [ ] No customer-facing public type or func signature in `pkg/` was
  removed silently. Silent removals are tier-2 escapes — push back.
- [ ] Facade `*Options` fields all correspond to api/ wire fields via
  their `optionsTo<X>` mapping. The facade-mirror guard pass should
  have caught hallucinated fields (e.g. auth fields on per-request
  options), but spot-check.
- [ ] All converters use the canonical deref helpers; no inline nil
  checks on primitive fields.
- [ ] Idiomatic Go casing for initialisms (`RequestID` not
  `RequestId`, `APIVersion` not `ApiVersion`) is applied to every new
  customer field.
- [ ] New union variants added in `api/` are reflected in the
  facade's `ConvertServerMessage` type switch with a corresponding
  customer event type.

### Agentic readiness (per [`sdk-agentic-readiness`](../sdk-agentic-readiness/SKILL.md))

- [ ] Every new public type or function in `pkg/` has an `Example_*`
  in the adjacent `*_test.go`.
- [ ] Every changed signature has its `Example_*` updated.
- [ ] `go test ./...` passes (this exercises every `Example_*`).
- [ ] `README.md` opening paragraph still keyword-dense.
- [ ] `llms.txt` still points at valid files.
- [ ] No multi-concept files appeared (no `utils.go` / `helpers.go`,
  no files mixing REST and WebSocket types).

### Breaking ceremony

- [ ] If the PR has a `regen/breaking-absorbed` label (tier 1), the
  bot wrote an entry in `BREAKING_CHANGES.md`. Read it. Decide if the
  absorption is acceptable.
- [ ] If the PR has a `regen/breaking-unavoidable` label (tier 2),
  there is a `BREAKING_CHANGES.md` entry and the
  `spec-idiomatic/breaking-acked` status check is failing. You must
  explicitly apply the `regen/breaking-acked` label after deciding
  the break is acceptable.
- [ ] If there are no breaking labels, there should be no
  `BREAKING_CHANGES.md`. If one is present without a label, that's a
  bot bug — push back.

### Build & test

- [ ] `go build ./...` green.
- [ ] `go test ./...` green (covers `Example_*` smoke tests).
- [ ] No new TODOs or `panic("not implemented")` in `pkg/`.

## Signs the bot got it wrong

- Multiple retry attempts in the workflow log (`attempt 2/3`,
  `attempt 3/3`). The bot is allowed up to N retries; if it landed
  late the facade edits may be precarious. Read them carefully.
- A breaking entry in `BREAKING_CHANGES.md` that doesn't match the
  api/ diff. Push back; ask for a re-run.
- A facade edit that introduces a NEW exported type whose name doesn't
  match the established naming. The bot should be reusing existing
  customer types; new ones need human design intent.
- A facade `*Options` field added that doesn't correspond to any api/
  wire field — the facade-mirror guard should have caught it. If you
  see one in the diff, the guard pass failed; push back.
- An `Example_*` that uses `// Output: ...` for non-deterministic
  output. Push back; ask for a compile-only example.
- The bot deleted an example. Examples are load-bearing for retrieval
  scores; deletions should be tier-2 breaking-acked, not silent.
- The bot wrote `BREAKING_CHANGES.md` directly (no cleanup-pass
  commit). The LLM should never author this file; the deterministic
  pass owns it. Push back.

## When to push back

You should push back (close-without-merging or request changes) when:

1. Customer-facing public surface was broken silently (tier-2 escaped
   ceremony).
2. The facade absorption produced a customer field that's always nil
   when it shouldn't be.
3. The bot reached `max-retries` and the final state has broken
   tests.
4. A multi-concept file appeared.
5. `BREAKING_CHANGES.md` doesn't actually describe the breaking change
   accurately.
6. A facade field appeared that doesn't have an api/ counterpart.

In all six cases, the right move is to ping `#team-devrel-internal`
with the PR link and the specific concern. Don't merge a half-broken
regen to "fix in a follow-up" — the next regen will rebuild on top of
it.

## When to ack and merge

Everything in the checklist is green, breaking labels (if any) have
been acked, and you've read the facade diff with your own eyes. Merge.
