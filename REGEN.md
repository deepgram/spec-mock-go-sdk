# REGEN — Regeneration guide for maintainers

This document is for maintainers with access to the upstream tooling that produces this SDK. If you just want to use the SDK, see [`README.md`](README.md).

## What this repo contains

Two layers:

| Directory | Content | Edit policy |
|---|---|---|
| `api/` | Machine-generated wire types and transport primitives. Every file begins with `// Code generated by ... DO NOT EDIT.`. | Touching anything here will be overwritten on the next regeneration. |
| `pkg/` | Hand-written idiomatic Go client code, kept in lock-step with `api/`. Customer call signatures live here and are designed to stay stable across upstream changes. | Edit freely. |

Plus the standard top-level files (README, examples, go.mod, etc.). `examples/` is customer-facing runnable code; `pkg/.../example_test.go` files supply per-symbol godoc examples that double as compile-checked tests.

## How a regeneration lands

The upstream tooling watches for changes to the source-of-truth wire definitions. When a change merges, the tooling:

1. Opens a PR on this repo with `api/` updated to match the new wire definitions.
2. Runs an automated pass that reconciles `pkg/` against the new `api/` shape, pushing any required edits onto the same PR branch.
3. Posts a checklist comment summarising what changed at each tier (additive, absorbed-with-ceremony, unavoidable-breaking).
4. A human reviewer reads the diff, acknowledges any breaking ceremony, and merges.

The reviewer's job is to verify the automated pass made the right call. The PR-review checklist that follows is the canonical script.

## PR-review checklist

For any auto-regeneration PR:

- [ ] The PR head is an automated regen commit. If a human pushed on top of it, read every human commit individually.
- [ ] `api/` changes look like upstream wire changes (added / removed / renamed fields). No hand-edits.
- [ ] Every option field on a `*Options` struct in `pkg/` corresponds to a wire field in `api/`. A field on the client side without a wire counterpart is a hallucination and should be removed.
- [ ] Every removed wire field has been absorbed by the client at the right tier:
  - **Additive (tier 0)** — new field on `*Options` plus wiring. Silently absorbed.
  - **Absorbed-with-ceremony (tier 1)** — wire field removed but the field stayed on `*Options` (marked deprecated), wiring dropped, `TestDropped_<Field>` replaces `TestWires_<Field>`, `BREAKING_CHANGES.md` entry written.
  - **Unavoidable-breaking (tier 2)** — wire field removed AND `*Options` field removed. Customers get a build break. `BREAKING_CHANGES.md` entry, orphaned wire tests cleaned up.
- [ ] Any new exported type's name reads as customer-facing (`Connect`, `Send`, `Recv`, ...). Internal architectural names should never appear in `pkg/`.
- [ ] Wire tests in `pkg/.../wire_test.go` still pass and the diff matches the field changes above.
- [ ] Runnable godoc examples in `pkg/.../example_test.go` still compile.
- [ ] `go test ./...` and `go build ./...` are green.

If anything in this checklist looks off, leave a review comment on the PR rather than merging. The automated pass is a starting point, not a finished product.

## Triggering a regeneration manually

Most of the time you don't need to. Wait for the upstream watcher to open the PR. If you do need to run it from your laptop (testing an unmerged upstream change, debugging a regen issue), the prerequisites and step-by-step are documented internally in the regeneration tooling itself — start there.

## Build and test locally

```bash
go build ./...
go test ./... -count=1
```

Both must pass for an auto-regen PR to land. `go test` runs every `Example_*` function in the package; if a runnable example stops compiling, the build fails.

## Vocabulary

This repo is — or will become — public. Internal Deepgram architectural terms must not appear in any file: code, comments, docs, example READMEs, llms.txt, or commit messages. When in doubt, use generic English. Customer surface uses customer vocabulary.

## What changes when

| You changed | What needs to happen |
|---|---|
| A wire definition upstream | Wait for the automated PR. Review per the checklist above. |
| A `pkg/` file by hand | Run `go test ./...`. If you added a new public symbol, add a matching `Example_*` test. If you added a new field to `*Options`, verify a wire test covers it. |
| A new public symbol | Add a godoc example block referencing it. Add an `Example_<Symbol>` function in the adjacent `example_test.go`. Re-run `go test ./...`. |
| README, examples, or llms.txt prose | Verify it doesn't contradict the code. The source is the substrate; prose derives from it, never the other way around. |

If you find prose that contradicts the code, fix the prose. Not the code.
